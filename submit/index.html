Submission guide

Submit a tool with enough proof to review it.

Submissions start from a structured intake file, CI validation, and a human review pass. The public queue only mirrors safe status updates after review.

Weekly review cycleProof requiredTwo submission paths

Qualification

What qualifies for review

A new visitor should be able to install and run the tool, understand its limits, and inspect at least one proof link.

Our worthiness rubric A checklist that scores tools on documentation, tests, releases, and other quality signals. requires at least 3/5.

  • License is OSI-approved
  • At least 1 release published
  • README has install + usage
  • Activity within last 90 days
  • No known security issues

Disqualifiers

What gets rejected quickly

  • No install path or unclear runtime requirements.
  • "Coming soon" repos with no working code.
  • No license or claims with no evidence.
  • Tools already present in the catalog.

Submission flow

How to submit

  1. Fork the marketing repo.
  2. Create submissions/<your-slug>.json with the public submission metadata, including submission.lane.
  3. Open the promotion or experiment pull request template and wait for CI validation.
  4. CI projects the public queue row into site/src/data/submissions.json; maintainers patch only review status there.

Submission schema

Required shape

{
  "tool": {
    "name": "My Tool",
    "slug": "my-tool",
    "repo": "https://github.com/org/my-tool"
  },
  "category": "devtools",
  "kind": "cli",
  "pitch": "A short description of what the tool does and why it matters.",
  "goodFor": [
    "Use case one",
    "Use case two"
  ],
  "notFor": [
    "Not intended for this"
  ],
  "proof": [
    {
      "label": "CI",
      "url": "https://github.com/org/my-tool/actions",
      "whatItProves": "Tests pass"
    }
  ],
  "install": "npm install -g my-tool",
  "quickstart": "my-tool init",
  "maintainer": {
    "handle": "@you"
  },
  "submission": {
    "lane": "promo",
    "submittedAt": "2026-03-17T09:00:00Z",
    "pr": "https://github.com/org/marketing-repo/pull/42"
  }
}

Two lanes

Choose a submission path

Promotion

Tools that clear the rubric move into the public promotion pipeline with proof pages, press coverage, and receipts.

Experiment

Earlier tools can enter the experiment lane for controlled copy testing before they graduate to a public push.

Review policy

What happens after CI

  • Submissions are reviewed on a weekly cadence.
  • Entries may be marked needs info while details are clarified.
  • The public queue shows only status-safe updates, not reviewer-only notes or internal escalation details.
